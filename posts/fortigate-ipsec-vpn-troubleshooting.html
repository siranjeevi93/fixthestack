FortiGate IPSec VPN Tunnel Drops — RCA & Fix

fortigate ipsec vpn fortinet

Phase 1 was stable. ISAKMP handshake completed fine every time. But Phase 2 — the actual data tunnel — was flapping every 3–4 hours like clockwork. Users were screaming, and get vpn ipsec tunnel summary was showing the tunnel in down state right after it came back up.

Here's how I traced it, what was actually wrong, and what fixed it permanently.

Environment

  • FortiGate 200F running FortiOS 7.2.x on both ends
  • Site-to-site IPSec tunnel — HQ to branch office
  • Traffic: mixed TCP (RDP, SMB) + UDP (VoIP)
  • ISP at branch: 4G failover kicking in on primary link drops

Initial Investigation

Step 1 — Check Phase 2 lifetime mismatch

First thing I always check: are Phase 2 lifetimes identical on both peers? A mismatch causes one side to try rekeying while the other isn't ready. 

FortiGate CLI
config vpn ipsec phase2-interface
    edit "branch-p2"
        get
    next
end

Both sides showed keylifeseconds: 43200 (12h) — matched. Not the issue.

Step 2 — Check DPD (Dead Peer Detection) settings

DPD is the keepalive mechanism. If one peer sends DPD probes and doesn't get a reply (even temporarily, due to transient packet loss), it tears down the tunnel. 

FortiGate CLI
config vpn ipsec phase1-interface
    edit "branch-p1"
        show | grep dpd
    next
end

Output showed:

set dpd on-idle
set dpd-retrycount 3
set dpd-retryinterval 20
The problem: on-idle mode sends DPD probes only when there's no traffic. But with retrycount 3 and retryinterval 20s, if the branch ISP hiccupped for just 60 seconds during off-hours (low traffic = idle), FortiGate would declare the peer dead and tear down Phase 2. The 4G failover was taking ~90 seconds to come up — just long enough to miss all DPD retries.

Step 3 — Confirm with FortiAnalyzer logs

Pulled the IPSec event logs from FortiAnalyzer for the exact drop times: 

FortiAnalyzer log filter
type=event subtype=vpn action=tunnel-down

Every drop had the reason: DPD failure - peer not responding. Timestamp correlated exactly with branch ISP outage windows logged in the router.

The Fix

Option 1 — Increase DPD retry tolerance

FortiGate CLI — both peers
config vpn ipsec phase1-interface
    edit "branch-p1"
        set dpd on-idle
        set dpd-retrycount 10
        set dpd-retryinterval 30
    next
end

This gives the tunnel 300 seconds (5 min) to recover before DPD kills it — enough time for the 4G failover to complete and traffic to resume.

Option 2 — Switch to on-demand DPD (preferred)

FortiGate CLI — both peers
config vpn ipsec phase1-interface
    edit "branch-p1"
        set dpd on-demand
    next
end
on-demand only sends DPD probes when there's traffic to send but the tunnel appears unresponsive. This is far less aggressive than on-idle for intermittent link scenarios. We switched to this and the random drops stopped immediately.

MTU — the other culprit

After fixing DPD, we still saw occasional large-file transfer failures (SMB copies stalling at 30-40%). This is almost always MTU fragmentation.

IPSec adds overhead: ESP header (8 bytes) + IV + padding + auth tag. On a 1500 MTU link, the effective MTU inside the tunnel is around 1418 bytes. If the inner packet is larger and the DF (Don't Fragment) bit is set, packets silently drop. 

FortiGate — set MSS clamp
config system interface
    edit "branch-p1"
        set tcp-mss 1350
    next
end
Result: SMB transfers stabilized. No more silent drops on large files.

Summary — What to Check for IPSec Drops

  1. Phase 1 & 2 lifetime mismatch → match them exactly on both peers
  2. DPD mode and timers → use on-demand, increase retrycount for flaky links
  3. MTU / fragmentation → set tcp-mss to account for IPSec overhead
  4. NAT-T → if either peer is behind NAT, enable set nattraversal enable
  5. FortiAnalyzer → always pull tunnel-down events and correlate with ISP logs
