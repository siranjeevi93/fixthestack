Building a Zero-Database Website Uptime Monitor — with an MCP Server Built In

mcp python monitoring uptime

I wanted a way to keep an eye on a handful of internal and external sites — some behind self-signed certs, one or two with no DNS record at all — without standing up a database, a message broker, or paying for a SaaS uptime monitor. So I built Website Checker: a pure-Python monitor with three faces — an MCP server, an hourly cron checker, and a web dashboard — all sharing the same flat-file state.

Why no database

State is just JSON on disk: sites.json, status.json, and an append-only history.jsonl. That sounds primitive, but it means the whole thing is trivial to deploy, back up (it's just files), and reason about. There's exactly one module, monitor_core.py, that owns the check logic and the store — every other entry point calls into it instead of duplicating logic.

Three ways in, one source of truth

  • MCP server (server.py) — add/remove sites and query results from Claude Code, Claude Desktop, or any MCP client over stdio.
  • Hourly checker (monitor.py) — a cron-driven sweep that records uptime, latency, and status codes for every configured site.
  • Web dashboard (dashboard.py) — a Flask UI on :8090 with separate Internal / External panes and live status.

Because all three read and write the same JSON store, the dashboard always reflects the most recent sweep — no cache invalidation, no coordination layer, nothing to get out of sync.

The edge cases that actually mattered

Self-signed / private-CA hosts

Internal appliances (vCenter, .local hosts) often carry self-signed certs. A hard TLS-verify would flag them as down for the wrong reason. Each site supports a per-site verify_tls=False so cert-trust failures don't masquerade as outages.

Hosts with no DNS record

Some internal targets aren't in DNS at all. Rather than editing /etc/hosts on the monitor box, a per-site resolve_ip option connects directly to a fixed IP while still sending the real hostname in the Host header — so the health check exercises the right vhost without touching system config.

Alerting without a mail relay

I didn't want to depend on Postfix or an SMTP relay just to get a down-alert email. Instead, alerts.py delivers direct to the recipient's MX over SMTP port 25 with opportunistic STARTTLS. When a sweep finds sites down, it sends one consolidated alert (repeated hourly while they stay down); when a site recovers, it sends a one-time recovery notice. The whole feature is opt-in — it's disabled whenever WM_ALERT_TO is empty, so the repo itself never carries real addresses.

Health rule: a site counts as healthy when it returns HTTP < 400 within the timeout (10s by default). Timeouts, connection errors, and 4xx/5xx all count as down.

Quick start

bash
git clone https://github.com/siranjeevi93/website_checker.git
cd website_checker

python3 -m venv venv
./venv/bin/pip install -r requirements.txt

# verify everything works (uses an isolated temp store)
./venv/bin/python smoke_test.py

# run a one-off sweep, then launch the dashboard
./venv/bin/python monitor.py
./venv/bin/python dashboard.py      # http://localhost:8090

MCP tools exposed

Once registered with an MCP client, these are the tools available:

  • add_site(url, name, category, verify_tls, resolve_ip) — add or update a monitored site
  • remove_site(url_or_name) — stop monitoring a site
  • list_sites() — list everything currently configured
  • check_now(url=None) — run an immediate check, one site or all
  • status(url=None) — latest result per site
  • history(url=None, limit=50) — recent check history
  • uptime(url=None) — uptime % computed on demand from history
  • test_alert() — send a test alert email to configured recipients

What I'd change next

history.jsonl auto-trims, but for a fleet larger than a few dozen sites I'd eventually want something queryable — SQLite is the obvious next step without giving up the "just files" simplicity. For now, at the scale I'm running it, the flat-file store is one less thing to operate.
